# Identity Verification

**Identity Verification** lets Business Actions declare a minimum authentication level. The runtime enforces that level before any outbound API call — Business Actions never contain WhatsApp-, widget-, or portal-specific auth logic.

## Authentication levels

| Level | Meaning | Typical sources |
| --- | --- | --- |
| `anonymous` | Public visitor | Widget without `identify()` |
| `verified_phone` | Phone ownership established | WhatsApp Meta-verified sender |
| `strong` | Strong end-user proof | Host JWT/`identify()`, portal login, Email OTP session |

Higher levels satisfy lower requirements (`strong` ⊇ `verified_phone` ⊇ `anonymous`).

## Business Action policy

Set on each tool:

```json
{ "required_auth_level": "strong" }
```

Values: `anonymous` (default), `verified_phone`, `strong`.

When the caller’s level is insufficient, the runtime returns an **auth challenge** (structured like a precondition failure) and does **not** call your API.

## Phone mappings

Local table (Admin API):

- `PUT /api/v1/workspaces/:id/phone-mappings` — upsert `{ phone, customer_id?, email?, display_name? }`
- `GET /api/v1/workspaces/:id/phone-mappings`
- `DELETE /api/v1/workspaces/:id/phone-mappings/:id`

Email is required for Strong Email OTP on WhatsApp.

## Widget endpoints

- `POST /api/v1/widget/identity/challenge` — start Email OTP (`{ email, conversation_id? }`)
- `POST /api/v1/widget/identity/verify` — confirm (`{ challenge_id, code }`)
- Existing `identify()` JWT/session already counts as **Strong**

## Identity forwarding

Outbound Business Tool requests receive:

| Header | Example |
| --- | --- |
| `X-Qefro-User-ID` | customer / user id |
| `X-Qefro-Organization` | tenant UUID |
| `X-Qefro-Authentication-Level` | `verified_phone` / `strong` |
| `X-Qefro-Channel` | `whatsapp` / `widget` / `portal` |
| `X-Qefro-Phone` | E.164 when known |

Plus existing `END_USER_IDENTITY` Bearer / session forwarding when configured.

## Security notes

<Warning>
OTPs are stored hashed (HMAC), single-use, rate-limited, and never written to audit logs. Verified sessions live in Redis for 20 minutes.
</Warning>

<InfoBox>
Authentication ≠ authorization. After the auth-level gate, existing tool preconditions (`require_authenticated`, permissions, variables) and workspace RBAC still apply.
</InfoBox>
