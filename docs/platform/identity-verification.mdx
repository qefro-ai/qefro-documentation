# Identity Verification

**Identity Verification** lets Business Actions declare how much trust they need. The runtime uses **channel-provided identity first** and only starts an organization challenge when you explicitly require it.

Business Actions never contain WhatsApp-, widget-, or portal-specific auth logic — adapters build a shared [`ChannelIdentity`](/docs/platform/identity-verification) / `VerifiedIdentity`.

## Authentication requirements

| Requirement | Meaning | When to use |
| --- | --- | --- |
| `public` | No identity required | FAQs, hours, pricing, general support |
| `verified_channel` | Trust already provided by the channel | Order status on WhatsApp; “my orders” on a logged-in website |
| `organization_challenge` | Extra org-defined verification | Invoices, medical records, bank changes, account deletion |

Configure on each tool:

```json
{ "required_auth_level": "verified_channel" }
```

Default: `public`. Legacy values `anonymous` / `verified_phone` / `strong` still parse for compatibility.

## Design philosophy

- Do **not** force OTP or login unless the organization sets `organization_challenge`.
- WhatsApp Meta-verified phone **is** verified channel identity — no OTP for `verified_channel`.
- Website Widget JWT / session from `identify()` **is** verified channel identity — no OTP for `verified_channel`.
- Qefro orchestrates organization challenges; your systems remain the authority for access to your APIs (see [Identity Forwarding](/docs/platform/identity-forwarding)).

## Channel identity

| Channel | Verified when | Identity type |
| --- | --- | --- |
| WhatsApp | Meta-verified phone present | `phone_number` |
| Website Widget | Host JWT or session via `identify()` | `jwt` / `session` |
| Internal Portal | Employee login (tools blocked in V1) | — |

If a tool requires `verified_channel` and the channel has no verified identity, the runtime returns **`CHANNEL_IDENTITY_REQUIRED`** (widget: ask the customer to sign in on your site). **No Email OTP** is started.

## Organization challenge

Only when:

```json
{ "required_auth_level": "organization_challenge" }
```

Runtime returns **`ORGANIZATION_CHALLENGE_REQUIRED`**. In v1 the orchestrated method is **Email OTP** (hashed, single-use, 20-minute verified session). Future methods: SMS OTP, magic link, OAuth, customer portal login.

Existing challenge context skips a repeat challenge until the session expires.

## Phone mappings

Local table (Admin API):

- `PUT /api/v1/workspaces/:id/phone-mappings` — upsert `{ phone, customer_id?, email?, display_name? }`
- `GET /api/v1/workspaces/:id/phone-mappings`
- `DELETE /api/v1/workspaces/:id/phone-mappings/:id`

Email is required to start Email OTP for `organization_challenge` on WhatsApp.

## Widget endpoints

- `POST /api/v1/widget/identity/challenge` — start Email OTP (`{ email, conversation_id? }`) — used for **organization_challenge**
- `POST /api/v1/widget/identity/verify` — confirm (`{ challenge_id, code }`)
- `identify()` JWT/session counts as **`verified_channel`** (not organization challenge)

## Identity forwarding

Outbound Business Tool requests receive:

| Header | Example |
| --- | --- |
| `X-Qefro-User-ID` | customer / user id |
| `X-Qefro-Organization` | tenant UUID |
| `X-Qefro-Authentication-Level` | `public` / `verified_channel` / `organization_challenge` |
| `X-Qefro-Channel` | `whatsapp` / `widget` / `portal` |
| `X-Qefro-Phone` | E.164 when known |

Plus existing `END_USER_IDENTITY` Bearer / session forwarding when configured. Tokens are never sent to the LLM.

## Security notes

<Warning>
OTPs are stored hashed (HMAC), single-use, rate-limited, and never written to audit logs. Verified sessions live in Redis for 20 minutes. Never trust client-asserted identity without channel validation.
</Warning>

<InfoBox>
Authentication ≠ authorization. After the auth gate, tool preconditions (`require_authenticated`, permissions, variables) and workspace RBAC still apply. Internal Portal Business Tool execution remains blocked in V1.
</InfoBox>
