# Business Tools Best Practices

## Rollout

1. **Read-only first** — `GET` status, search, list.
2. **One workspace** — Pilot customer support before HR/finance tools.
3. **Test Tool** — Every tool before chat enablement.
4. **Logs review** — First week of production traffic.
5. **Writes later** — Idempotency + confirmation for mutations.

## REST

- Scope API keys per tool/environment.
- Trim OpenAPI imports to assistant-safe operations.
- Use `END_USER_IDENTITY` only when your API validates end-user JWT.
- Return structured JSON with explicit `found` / error codes.
- Document rate limits — Qefro retries transient errors.

## SDK

- Keep handlers small; domain logic in services.
- Implement `lookup` and `authorize` separately.
- Use `lookup.required` instead of hardcoding channel checks.
- Mask PII in challenge hints.
- Re-sync after every handler/schema change deploy.

## Identity

| Scenario | Approach |
| --- | --- |
| Logged-in website user | `identify()` + REST `END_USER_IDENTITY` |
| Anonymous + email OTP | SDK `lookup: email` + authorize |
| WhatsApp | SDK `lookup: phone` |
| Never | Admin JWT in widget token field |

## Workspace hygiene

- Customer tools ≠ employee tools (separate workspaces).
- Disable unused tools instead of deleting during audits.
- Name tools for the LLM (`order_status_check` not `api1`).

## Mixed integrations

- Vendor APIs → REST/OpenAPI.
- Core customer auth → SDK.
- Document which tools require login in tool **description** (LLM reads it).

<Warning>
Never expose long-lived admin credentials to the browser. Service secrets stay in Qefro encrypted storage; end-user auth uses identify() or SDK challenges.
</Warning>
